 - 깃허브 사용법 Merge 와 conflict

//브렌치는 개발 구역을 나눠주기
저장소 이름: testpart
원격 저장소에 존재하는 브렌치 : main ,(추가) Develop

브랜치 확인
git branch

원격레포지토리에 브렌치 확인
git branch -r

git checkout -b 브랜치이름
-b로 브랜치를 생성하면서 이동해라

git checkout
그 브랜치로 이동해라

오리진(원격 저장소)에 너의 디벨롭 브렌치를 push해라
git push origin Develop

위에 원형을 써도 된다.
원격저장소에 up스트림을 일치시키기(브렌치로 예시 : Develop)
git branch --set-upstream-to=origin/Develop

//태그는 완성본? 혹은 어느 지점의 부분(버전별로?)을 올려주기 위해 사용
태그확인
git tag

태그 추가
git tag (태그이름)v0.0.1

태그 원격에 동기화,추가
git push (태그이름)v0.0.1

원격저장소와 로컬저장소를 일치시켜주기
git fetch -p

commit 로그 내역 가져오기 (커밋주소와 그에 따른 태그나 브렌치 명을 볼수 있음)
git log --all

git pull origin v0.0.1 ({{{{커밋 태그명, 브렌치명}}}를 통해 그때의 상태를 원격저장소에서 로컬 저장소로 가져온다)
(원격저장소랑 로컬 저장소 일치 화 후)
그후 git push 하면 적용된다. [원격저장소에 선언]

merge는 시간 순서대로 아래에서 위로 (트리) 
git merge origin/Develop
git merge 브렌치 이름
merge할때 같은 충돌이 나면 그 소스코드 파일에 들어가서 충돌 부분 수정 후 저장 ([브랜치명 Alpha]|MERGING) 충돌중인 상황
다시 git add . commit push
브렌치 삭제 전 항상 맞춰주기(git fetch -p)
git checkout 브렌치이름 (현재 브렌치를 지우기위해선 현재 위치에서 나와줘야한다)
git  branch -D 브렌치 이름 (삭제선언)
git push origin --delete 브렌치이름(삭제적용)

merge와 pull의 사용 차이
merge는 2칸이상의 차이 (뒤로 돌아갔다 앞으로 한칸)같은 차이가 나면 써주고 이후 merge가 되면 1칸의 차이가 나니
그때 한칸 차이가 날때 pull을 이용해 받고 push로 밀어준다

merge 순서
브렌치를 합치고 나중에 브렌치 지우기

사용이유
main 은 남겨두고 (develop같은 다른 브렌치) 계속 다른 걸로 merge 하다가 나중에 main 올리기

잘못되면 tag 들을 통해 복구한다.
